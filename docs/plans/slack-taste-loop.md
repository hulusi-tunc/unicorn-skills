# Plan: Slack taste loop

**Status:** Tasks 1–2 built · **Architecture:** Slack MCP pull (no infrastructure) · **Owner:** Hulusi

## What this is

A taste-capture system for the design team. Anyone posts an image (with a comment) to `#design-inspiration` in Slack; `/taste-pull` files it into a versioned taste library in this repo with the poster's own words and an attribute analysis. `/taste-sync` distills the library into `taste/TASTE.md` — a living profile of what the team likes and dislikes. Claude Code reads that profile when designing: the `no-slop` and `taste` skills point every visual-direction decision at it.

The loop: **post in Slack → pulled + analyzed + filed → synthesized into a profile → shapes the next design → react 👍/👎 in Slack → profile sharpens.** Every save and every reaction is training signal.

## Architecture

```
Slack #design-inspiration  (C0BFEC85BU4)
  │  message + image (+ 👍/👎 reactions)
  ▼
/taste-pull  — Slack MCP connector, runs inside Claude Code
  │  slack_read_channel since last_ts → slack_read_file (base64 image)
  │  → vision analysis → entry committed to taste/library/ + taste/images/
  │  → ✅ reaction on the filed message → slack_get_reactions refreshes scores
  ▼
/taste-sync  — regenerates taste/TASTE.md (weighted: score × 90-day half-life)
  │  optional: posts a ≤5-line digest back to the channel
  ▼
taste skill  — unicorn / ui-designer / designer-copilot read TASTE.md
              before proposing any visual direction; grep library for references
```

Why this shape: the session already has an authenticated Slack MCP connection, so there is **no bot, no webhook endpoint, no tokens, no hosting**. Git is the database — versioned, reviewable, readable by Claude Code in every session. Pull is idempotent: `taste/slack.json` tracks `last_ts`, so `/taste-pull` always catches up from wherever it left off.

## Data model

```
taste/
  TASTE.md              # synthesized profile — regenerated by /taste-sync only
  slack.json            # { channel_id, channel_name, last_ts } — /taste-pull state
  library/              # one entry per saved item
    2026-07-06-a1b2c3.md
  images/
    2026-07-06-a1b2c3.png
  inbox/                # manual drop folder for /taste-add (no Slack needed)
```

**Entry format** (`taste/library/<date>-<slug>.md`):

```markdown
---
id: 2026-07-06-a1b2c3
date: 2026-07-06
source: slack            # slack | manual
author: hulusi
type: screen             # screen | component | typography | palette | motion | illustration | layout
tags: [dashboard, dark, data-dense, mono]
score: 1                 # 1 + 👍 − 👎, refreshed by /taste-pull for 30 days
status: active           # active | anti | archived   (anti = score ≤ −1, "we dislike this")
image: ../images/2026-07-06-a1b2c3.png
slack_ts: "1783307073.644709"
---

## Their words
"love how calm this table is, no borders at all"   ← the Slack message, verbatim

## Analysis
- Layout: single-column data table, no vertical rules, 48px row height
- Type: mono numerals, 13px labels in caps with tracking
- Color: near-black surface, one desaturated accent, no gradients
- Mood: calm, technical, confident
## Steal this
Alignment and whitespace doing the separation work borders usually do.
```

**TASTE.md sections** — Strong preferences (≥5 weighted entries) · Emerging (2–4) · Anti-taste · Directions this maps to (`no-slop` vocabulary) · Aging out (no reinforcement in 90 days). Weight = score × recency (90-day half-life). Regenerated in full, never appended.

## Tasks

### Task 1 — taste library + Claude Code integration ✅ built
`taste/` scaffold, `taste` skill (wired into unicorn, ui-designer, designer-copilot), `/taste-add` (manual inbox flow), `/taste-sync` (synthesis). Works with no Slack at all.

### Task 2 — Slack pull ✅ built
`/taste-pull` command + `taste/slack.json` state, per the architecture above. `/taste-sync` gained the optional channel digest. Fallback when the Slack MCP isn't connected: `/taste-add`.

### Task 3 — live acceptance (needs real content)
1. Team posts 3+ real inspiration images with one-line comments to `#design-inspiration`.
2. Run `/taste-pull` → entries appear with verbatim words, sensible tags, ✅ reactions on the messages.
3. React 👎 on one message, `/taste-pull` again → its entry's score drops.
4. Run `/taste-sync` → TASTE.md reflects the entries; digest posts to the channel.
5. Ask Claude to "design a settings screen" → the direction cites the profile.

### Task 4 — habit + cadence (after Task 3 passes)
- Team ritual: whoever opens Claude Code first each day runs `/taste-pull`; `/taste-sync` weekly (Monday).
- Optionally automate with a scheduled cloud agent (`/schedule`) — verify first that the Slack connector is available in scheduled runs; if not, the manual ritual stands.
- Monthly: `/taste-sync` digest asks the team to confirm or kill "Aging out" items via reactions.

## Constraints

- The poster's words are never paraphrased — they are the highest-value signal.
- Only `/taste-sync` writes `TASTE.md`; only `/taste-pull` and `/taste-add` write library entries.
- Client brand always beats taste; taste breaks ties. Anti-taste is a warning, not a law.
- Every taste commit is named `taste: ...` so the library's history stays auditable.

## Appendix — real-time push (only if pull ever feels too slow)

The original v1 design: a Slack Events API bot on two Vercel functions (`/api/slack/events` + weekly cron), Claude vision per image (`claude-haiku-4-5-20251001`), synthesis on `claude-fable-5`, commits via GitHub Contents API. Needs a Slack app (scopes: `channels:history`, `files:read`, `reactions:read`, `chat:write`), signing-secret verification, and env vars on Vercel. Skip unless the team wants sub-minute capture — the MCP pull gives the same library with zero moving parts.
